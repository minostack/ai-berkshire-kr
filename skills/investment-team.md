# 투자 리서치 팀: 4개 에이전트 병렬 분석 프레임워크

$ARGUMENTS 에 대해 팀 기반 투자 리서치 분석을 수행합니다. Team 도구를 사용하여 실제 멀티 에이전트 병렬 리서치 팀을 구성합니다.

## 실행 흐름

### 1단계: 팀 구조 안내

사용자에게 아래 팀 구조를 보여주고 확인 후 시작합니다:

| 역할 | 담당 | 분석 프레임워크 |
|------|------|----------------|
| **team-lead** (본인) | 총괄 조정 · 결과 종합 · 최종 리포트 출력 | 4대가 종합 프레임워크 |
| **business-analyst** | 비즈니스 모델 & 경제적 해자 분석 | 단용핑 관점 |
| **financial-analyst** | 재무제표 & 밸류에이션 분석 | 버핏 관점 |
| **industry-researcher** | 산업 구조 & 경쟁 구도 분석 | 멍거 관점 |
| **risk-assessor** | 리스크 평가 & 경영진 판단 | 리루 관점 |

### 1.5단계: AI 리서치 편향 평가

팀 구성 전, 해당 기업의 "AI 연구 가능성" 등급을 사용자에게 먼저 안내합니다:

**정보 풍부도 등급** (리서치 전략 결정):
| 등급 | 특징 | 리서치 전략 조정 |
|------|------|----------------|
| A등급 (정보 충분) | 오랜 상장 이력, 애널리스트 커버리지 풍부 | 팀은 **반론 검증**과 **비공감대 관점**에 집중 — 시장 공감대와 일치하는 "정답 같은 헛소리" 출력 방지 |
| B등급 (정보 보통) | 상장 기간 짧거나 커버리지 제한적 | 각 에이전트의 추정 데이터에 신뢰도 표기 필수, team-lead 종합 시 "데이터 충분도" 명시 |
| C등급 (정보 부족) | 비인기 종목 / 신규 상장 / 신흥 시장 | 팀을 "제1원칙 모드"로 전환: 리포트 완성도보다 비즈니스 본질의 핵심 질문 몇 가지에 집중 |

**핵심 주의사항**: 자료 많음 ≠ 확실성 높음, 자료 적음 ≠ 확실성 낮음. AI 출력 신뢰도 ≠ 실제 투자 확실성. 확실성은 비즈니스 모델 자체에서 나오지, 자료의 양에서 나오지 않습니다.

등급 결과를 각 에이전트에게 전달하여 리서치 방식에 반영합니다.

### 1.75단계: 웹 검색 권한 사전 확인 (필수 · 에이전트 기능 저하 방지)

팀 구성 및 백그라운드 에이전트 시작 **전에** 반드시 웹 검색 권한이 허용되어 있는지 확인합니다.

**사전 확인이 필수인 이유**: 이 스킬은 `run_in_background: true`로 4개의 백그라운드 서브 에이전트를 시작하는데, **백그라운드 에이전트는 사용자에게 권한 확인 팝업을 띄울 수 없습니다**. `WebSearch`가 `.claude/settings.local.json`의 `permissions.allow` 화이트리스트에 없으면, 서브 에이전트의 인터넷 검색이 **무음으로 차단**되어 훈련 데이터(지식 마감일 있음)만으로 답변하면서 프레임워크에 맞춰 "완성된 것처럼 보이지만 실제로는 인터넷 미검색" 가짜 리서치를 출력하게 됩니다 (이것이 이 스킬의 가장 위험한 실패 패턴입니다).

**사전 확인 절차**:
1. Bash로 화이트리스트에 WebSearch가 있는지 확인:
   ```bash
   grep -l '"WebSearch"' .claude/settings.local.json ~/.claude/settings.local.json 2>/dev/null
   ```
2. 두 곳 모두 없으면 (미허용) → **멈추고 에이전트를 시작하지 말고** 사용자에게 안내:
   > ⚠️ WebSearch가 권한 화이트리스트에 없습니다. 백그라운드 리서치 에이전트가 인터넷에 접속할 수 없어 훈련 데이터만으로 답변하게 됩니다. `.claude/settings.local.json`의 `permissions.allow`에 `"WebSearch"`를 추가하거나 `/permissions`를 실행하여 체크한 후 다시 이 명령어를 실행해 주세요.
3. 확인되면 → 정상 진행.


### 1.76단계: 데이터 수집 도구 제한 사전 안내 (필수)

백그라운드 에이전트는 아래 사이트에 **Fetch로 직접 접속이 불가**합니다.
4개 에이전트가 시작되기 전에 이 제한을 반드시 인지하고 올바른 방법으로 데이터를 수집합니다:

| 차단 사이트 | 대체 방법 |
|-----------|---------|
| `finance.naver.com` (직접 Fetch) | `kstock_data.py quote/financials` 또는 WebSearch |
| `comp.fnguide.com` (직접 Fetch) | WebSearch로 대체 |
| `data.krx.co.kr` (직접 Fetch) | `kr_scanner.py` 또는 WebSearch |

**에이전트별 데이터 수집 우선순위**:
```
방법 1 (우선): python3 tools/kstock_data.py 또는 kr_scanner.py — DART API 직접 호출
방법 2 (보완): WebSearch 도구 — 검색엔진 경유로 차단 없이 수집
방법 3 (최후): DART 웹사이트 URL 안내 — 수동 확인 요청
❌ 절대금지: Fetch(finance.naver.com/...) 또는 Fetch(comp.fnguide.com/...)
```

각 에이전트 프롬프트에 이 제한 사항을 명시합니다.

### 1.8단계: 한국주 기본 데이터 자동 수집 (한국주인 경우 필수)

종목 코드가 6자리 숫자인 한국주는 팀 생성 전에 `kstock_data.py`로 기본 데이터를 먼저 수집합니다.
수집한 데이터는 4개 에이전트 프롬프트에 초기 컨텍스트로 포함합니다.

```bash
# 실시간 시세 및 밸류에이션 지표 수집
python3 tools/kstock_data.py quote {종목코드}

# 핵심 재무 데이터 수집 (DART API 키 등록 시 자동, 미등록 시 URL 안내)
python3 tools/kstock_data.py financials {종목코드}

# 밸류에이션 요약
python3 tools/kstock_data.py valuation {종목코드}
```

**수집 결과 활용:**
- `quote` 출력 → 재무 분석 에이전트(작업 2)의 현재 주가/시가총액/PER/PBR 초기값으로 사용
- `financials` 출력 → 재무 분석 에이전트(작업 2)의 최근 5년 재무 추이 초기값으로 사용
- `valuation` 출력 → `financial_rigor.py verify-market-cap` 검증의 입력값으로 사용

**수집 실패 시:** 도구 응답이 없으면 WebSearch로 DART(dart.fss.or.kr) 또는
네이버금융(finance.naver.com)에서 직접 수집하고 계속 진행합니다.

### 3단계: 4개 작업 생성

TaskCreate를 사용하여 아래 4개 작업 생성 (각각 subject, description, activeForm 포함):

#### 작업 1: 비즈니스 모델 분석
- subject: `{기업명}의 비즈니스 모델, 경제적 해자, 고객 가치 분석`
- description 포함 항목:
  1. 비즈니스 모델 본질: 핵심 사업 정의, 수익 구조 분해
  2. 플랫폼/제품 플라이휠 효과 작동 방식
  3. 해자 분석: 브랜드 / 전환 비용 / 네트워크 효과 / 규모의 경제 / 기술 장벽 — 각각 검증
  4. 고객/이용자 가치: 각 이해관계자에게 어떤 독자적 가치를 제공하는가
  5. 사업 포트폴리오와 시너지 효과
  6. 단용핑의 "좋은 사업" 기준 평가: 차별화, 가격 결정력, 지속 가능한 경쟁 우위
  7. 최신 재무제표, 산업 리포트 등 공개 정보 검색 필수

#### 작업 2: 재무 및 밸류에이션 분석
- subject: `{기업명}의 재무 데이터, 수익성 및 밸류에이션 분석`
- description 포함 항목:
  1. 최근 3~5년 매출, 순이익, 영업이익 추이
  2. 수익성 지표: ROE, ROA, 매출총이익률, 영업이익률
  3. 현금흐름 분석: 영업현금흐름, 잉여현금흐름, 자본적 지출
  4. 재무 건전성: 현금 보유액, 부채 비율, 유동성
  5. 밸류에이션 분석: PER/PSR/PBR/EV 등 — 역사적 수준 및 동종 업계 비교
  6. 안전마진 평가: 내재 가치 vs 현재 주가
  7. **재무 데이터 수집 및 정밀도 검증 (반드시 Bash로 도구 호출, 암산 금지)**:

     **[한국주] 데이터 수집 (1.8단계에서 수집한 값 활용, 없으면 직접 실행):**
     ```bash
     python3 tools/kstock_data.py quote {종목코드}       # 현재가/PER/PBR/외국인비중
     python3 tools/kstock_data.py financials {종목코드}  # 최근 5년 재무 데이터
     python3 tools/kstock_data.py valuation {종목코드}   # 밸류에이션 요약
     ```

     **[전 시장 공통] 재무 정밀도 검증:**
     - 시가총액 검증: `python3 tools/financial_rigor.py verify-market-cap --price {가격} --shares {주식수} --reported {보고된 시총} --currency {통화}`
     - 밸류에이션 검증: `python3 tools/financial_rigor.py verify-valuation --price {가격} --eps {EPS} --bvps {주당순자산}`
     - 핵심 데이터 교차 검증: `python3 tools/financial_rigor.py cross-validate --field {항목} --values '{JSON}' --unit {단위}`
     - 3가지 시나리오 밸류에이션: `python3 tools/financial_rigor.py three-scenario --price {가격} --eps {EPS} --shares {주식수억} --growth {낙관} {중립} {비관} --pe {낙관PE} {중립PE} {비관PE}`
     - 도구 출력 결과를 리포트에 직접 삽입하여 검증 기록으로 활용

#### 작업 3: 산업 및 경쟁 분석
- subject: `{산업} 산업 구조와 {기업명}의 경쟁 포지션 분석`
- description 포함 항목:
  1. 산업 규모와 성장: 시장 규모, 성장률, 침투율
  2. 경쟁 구도: 주요 경쟁사 시장 점유율, 경쟁 전략 비교
  3. 핵심 경쟁자 위협 평가: 주요 경쟁사별 개별 분석
  4. 세부 분야별 경쟁 구도
  5. 산업 트렌드: 기술 변화, 정책 영향, 신규 진입자
  6. 가치사슬 분석: 상류/중류/하류 가치 배분
  7. 최신 산업 데이터 및 경쟁 동향 검색 필수

#### 작업 4: 리스크 및 경영진 평가
- subject: `{기업명}의 투자 리스크와 경영진 역량 평가`
- description 포함 항목:
  1. 경영진 평가: CEO 역량 범위, 신뢰도, 전략적 비전, 자본 배분 능력, 과거 의사결정 품질
  2. 규제 리스크: 현재 및 잠재적 규제 영향
  3. 경쟁 리스크: 각 경쟁사별 위협 수준 평가
  4. 사업 리스크: 신규 사업 적자, 확장 불확실성
  5. 거시 리스크: 경기 사이클, 산업 사이클 영향
  6. 지배구조: 지분 구조, 특수관계인 거래, 주주환원 정책
  7. 장기 확실성: 10년 후 이 기업은 어떤 모습일까? 무엇이 비즈니스 모델을 무너뜨릴 수 있을까?
  8. 최신 규제 동향, 경영진 발언 등 검색 필수

### 4단계: 4개 병렬 에이전트 시작

Task 도구를 사용하여 4개 에이전트를 동시에 시작합니다 (**반드시 같은 메시지에서 병렬 호출**):

각 에이전트 설정:
- `subagent_type`: `general-purpose`
- `run_in_background`: `true`
- `team_name`: 해당 팀 이름
- `name`: 해당 역할명 (business-analyst / financial-analyst / industry-researcher / risk-assessor)

각 에이전트 프롬프트 템플릿:

```
당신은 {기업명} 투자 리서치 팀의 "{역할 한국어명}"으로, {대가명}의 투자 관점에서 {기업명}을 분석합니다.

작업 #{작업번호} 수행: {작업 subject}

세부 요구사항:
{작업 description 내용}

**리서치 방법**:
- **데이터 수집 도구 우선순위 (반드시 준수)**:
  - 1순위: `python3 tools/kstock_data.py financials {코드}` (한국주, DART API)
  - 2순위: WebSearch 도구 (검색엔진 경유 — 차단 없음)
  - 3순위: DART URL 안내 후 수동 확인
  - ❌ 절대 금지: `Fetch(finance.naver.com/...)`, `Fetch(comp.fnguide.com/...)` — 차단됨
- **재무 데이터는 반드시 2개 이상의 독립적인 출처에서 교차 확인**, `skills/financial-data.md` 기준에 따라 실행 (미국주: macrotrends+stockanalysis; 홍콩주: aastocks+macrotrends; 한국주: DART API+WebSearch), 두 출처 간 오차 >1%이면 표시
- 데이터 정확성 확보, 핵심 데이터에 출처 표기
- 표면적 분석이 아닌 심층 분석
- **인터넷 접속 실패 시 위장 금지**: WebSearch가 차단/불가능한 경우, 훈련 데이터를 인터넷 검색 결과처럼 가장하는 것을 금지합니다. 반드시 리포트 상단에 「⚠️ 이 리포트는 인터넷 미검색 상태로, 훈련 데이터(지식 마감일: X) 기반입니다. 신뢰도 하향 조정」을 명시하고 team-lead에게 알려 리서치 중단 여부를 결정하게 합니다.

**출력 요구사항**:
- 상세한 리포트, 핵심 데이터는 마크다운 표로 제시
- 각 분석 항목에 명확한 결론과 점수 포함
- 리포트 말미에 해당 분야의 종합 결론 포함

**완료 후**:
1. TaskUpdate로 작업 #{작업번호}를 completed로 표시
2. SendMessage로 전체 분석 리포트를 team-lead에게 전송 (type: "message", recipient: "team-lead")
```

| 에이전트 | 담당 분야 | 상태 |
|---------|---------|------|
| business-analyst | 비즈니스 모델 | ⏳ 분석 중 |
| financial-analyst | 재무 및 밸류에이션 | ⏳ 분석 중 |
| industry-researcher | 산업 및 경쟁 | ⏳ 분석 중 |

### 6단계: 팀원 종료

모든 리포트를 받은 후 4개 에이전트에게 shutdown_request를 전송합니다 (SendMessage, type: "shutdown_request").

### 7단계: 최종 리포트 종합

4개 분석 리포트를 종합하여 아래 구조의 최종 리포트를 출력합니다:

---

#### 1. 한 줄 결론
> 50~100자로 투자 가치 여부와 핵심 논리를 요약합니다

#### 2. 4개 분야 종합 평점표
| 분야 | 프레임워크 | 평점(1~5점) | 핵심 판단 |
|------|----------|-----------|---------|

종합 점수: X / 5

#### 3. 핵심 데이터 요약
주요 재무 및 영업 지표 표 (최근 2년 비교)

#### 4. 분야별 분석 요약
각 분야에서 가장 중요한 발견 3~5가지 정리

#### 5. 투자 논거 (강세 vs 약세)
- 🟢 매수 논리 (5~7가지)
- 🔴 매도 논리 (5~7가지)

#### 6. 버핏 매수 전 체크리스트
| # | 확인 항목 | 통과? | 설명 |
핵심 체크 항목 10가지 개별 평가

#### 7. 최종 투자 의견
- 정성적 판단표 (사업 품질 / 경영진 / 밸류에이션 / 타이밍)
- 유형별 매매 의견표 (공격형 / 안정형 / 보수형 → 의견 + 가격 범위)
- 핵심 트리거 (추가 매수 신호 / 매도 신호 각 3~5가지)

#### 8. 종합 마무리
100~200자 최종 요약

---

### 8단계: 리포트 저장

완성된 최종 리포트를 `reports/{기업명}투자연구보고서_{날짜}.md` (날짜 형식: YYYYMMDD)로 저장합니다.

### 9단계: 데이터 검수 (출고 전 검증)

```bash
# Step 1 — 검수 목록 추출 (15% 무작위 샘플링)
python3 tools/report_audit.py extract \
  --report <리포트 파일 경로>

# Step 2 — 목록 각 항목을 신뢰할 수 있는 출처에서 데이터 확인 (skills/financial-data.md 참조)

# Step 3 — 출고 승인 / 반려 판정 출력
python3 tools/report_audit.py verdict \
  --results '<완성된 JSON>' \
  --report <리포트 파일명>
```

**【출고 승인】** 전체 통과 → 리포트 발행 가능; **【반려】** 미통과 항목 있음 → 수정 후 재검수.

### 10단계: 팀 정리

TeamDelete로 팀 리소스를 정리합니다.

## 주요 주의사항

1. **4개 에이전트는 반드시 병렬 시작** — 같은 메시지에서 Task 도구를 4번 호출합니다
2. **에이전트는 SendMessage로 보고** — 파일 협업이 아닌 메시지 통신 방식입니다
3. **데이터 정확성** — 에이전트가 WebSearch로 최신 데이터를 검색하도록 요구하고, 핵심 데이터는 교차 검증합니다
4. **명확한 결론** — 매수 / 관망 / 회피 의견과 구체적인 가격 범위를 회피하지 않고 제시합니다
5. **모든 분석에 데이터 근거 필수** — 데이터 출처를 함께 첨부합니다
6. **충분히 기다리기** — 4개 에이전트의 리서치에는 몇 분이 소요됩니다. 실시간으로 사용자에게 진행 상황을 업데이트하세요
7. **편향 방지 의식** — team-lead는 종합 시 반드시 평가합니다: 각 에이전트의 분석이 정보 충분도에 의해 제한되지는 않았는가? 시장 공감대와 지나치게 일치하지는 않는가? 최종 리포트에 "정보 풍부도 등급"과 "AI 리서치 한계 안내"를 포함합니다
8. **정보 부족 시 솔직함 원칙** — 프레임워크를 채우기 위해 추측으로 메우는 것보다 "데이터 부족"으로 공백을 남기는 것이 낫습니다
